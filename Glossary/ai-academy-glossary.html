AI Foundations

LLM (Large Language Model)

▸
AI trained on massive text datasets to predict and generate human language.

Examples include GPT-5, Claude 4.5, Gemini 3.0, and Llama 4. LLMs form the core of modern AI systems and are the foundation for agents, chatbots, and content generation.

Token

▸
A unit of text (~4 characters or 0.75 words) that LLMs process.

LLMs don't read words directly — they split text into tokens. Pricing, context limits, and speed are all measured in tokens. A 128K context window means ~96,000 words.

Context Window

▸
The maximum amount of text an LLM can process in a single interaction.

Measured in tokens. Larger windows allow processing longer documents but increase cost and latency. Current models range from 8K to over 1M tokens.

Transformer

▸
The neural network architecture underlying all modern LLMs.

Introduced in the 2017 paper "Attention Is All You Need." Uses self-attention mechanisms to process text in parallel rather than sequentially, enabling much larger models.

Attention

▸
Mechanism allowing transformers to weigh the relevance of different parts of input text.

Self-attention lets the model focus on relevant words regardless of distance in the text. Multi-head attention runs multiple attention computations in parallel.

Inference

▸
Using a trained model to generate outputs, as opposed to training it.

When you send a prompt to an LLM and get a response, that's inference. Inference costs (compute, latency) are a key factor in production deployments.

Pre-training

▸
The initial phase of training an LLM on massive text datasets.

Involves processing billions of text samples to learn language patterns, facts, and reasoning. Extremely expensive (millions of dollars in compute). Done once by the model provider.

Fine-tuning

▸
Adapting a pre-trained model on specific data to specialize its behavior.

Uses a smaller, curated dataset to adjust model weights. Common for domain-specific applications (medical, legal, financial). Much cheaper than pre-training.

RLHF (Reinforcement Learning from Human Feedback)

▸
Training technique that aligns model outputs with human preferences.

Human raters rank model outputs, and the model is trained to prefer higher-ranked responses. Key technique behind making LLMs helpful, harmless, and honest.

Temperature

▸
Parameter controlling randomness of model outputs (0 = deterministic, 1+ = creative).

Low temperature produces consistent, focused responses ideal for factual tasks. High temperature introduces variety, useful for creative applications. Most production systems use 0–0.3.

Streaming

▸
Receiving LLM output token-by-token as it's generated, rather than waiting for the full response.

Provides faster perceived response times. Essential for chat interfaces. Requires handling partial responses in the UI.

Zero-shot

▸
Prompting a model to perform a task without providing any examples.

Relies on the model's pre-training knowledge. Contrasts with few-shot (providing examples) and fine-tuning. Works well for common tasks but may struggle with domain-specific ones.

AI Agents & Patterns

Agent

▸
An LLM combined with tools and goal-directed behavior, capable of taking autonomous actions.

Agents go beyond simple chat — they can call APIs, run code, search databases, and make decisions. The shift from "AI that talks" to "AI that acts."

Autonomous Agent

▸
An agent that operates with minimal human oversight, making decisions independently.

Can plan multi-step tasks, handle errors, and adapt to new situations. Requires robust guardrails and monitoring. Used in production for routine operations.

Multi-Agent System

▸
Architecture where multiple specialized agents collaborate to solve complex tasks.

Each agent has a focused role (researcher, coder, reviewer). Enables division of labor and specialization. Coordination patterns include orchestration, choreography, and swarm.

Orchestration

▸
A central controller directing and coordinating multiple agents.

The orchestrator decides which agent handles which subtask, manages data flow, and aggregates results. Common in enterprise systems where predictable behavior is required.

Tool Use

▸
An agent's ability to call external functions like APIs, databases, or services.

Tools extend what an LLM can do beyond text generation. Examples: web search, code execution, database queries, sending emails. Defined via function schemas.

Handoff

▸
When one agent transfers control or context to another agent in a multi-agent system.

Requires passing relevant context and state. Critical for complex workflows where different agents specialize in different domains.

ReAct (Reasoning + Acting)

▸
Pattern where agents alternate between thinking and taking action.

The agent thinks about what to do, executes an action, observes the result, then reasons about next steps. Produces more reliable results than acting without reasoning.

Planning Agent

▸
An agent that creates a structured plan before executing, breaking complex tasks into steps.

Plans can be revised based on intermediate results. Useful for multi-step tasks where order matters. Can include dependency tracking and parallel execution.

Reflection

▸
An agent evaluating its own output and iterating to improve quality.

The agent reviews its work, identifies issues, and corrects them before presenting the final result. Improves accuracy but adds latency and cost.

Memory

▸
An agent's ability to retain and recall information across conversations or sessions.

Short-term memory holds current conversation context. Long-term memory persists across sessions using databases. Essential for personalization and continuity.

MCP (Model Context Protocol)

▸
Open standard protocol for connecting AI models to external data sources and tools.

Created by Anthropic. Provides a standardized way for agents to discover and use tools, replacing custom integrations. Analogous to USB for AI tools.

Function Calling

▸
LLM capability to output structured calls to predefined external functions.

The model doesn't execute functions directly — it outputs a JSON specification of which function to call with what parameters. The application layer executes the actual call.

Agentic Workflow

▸
A process where AI agents autonomously perform multi-step tasks with decision-making.

Contrasts with simple prompt-response patterns. Includes planning, tool use, error handling, and iteration. The foundation of enterprise AI automation.

Prompt Engineering

Prompt

▸
The input text given to an LLM to guide its response.

Can range from a simple question to complex multi-part instructions. Prompt quality directly affects output quality. The primary interface between humans and LLMs.

System Prompt

▸
Hidden instructions that define an LLM's role, behavior, and constraints.

Set by the developer, not visible to the end user. Establishes personality, knowledge boundaries, output format, and safety rules. The "DNA" of an AI application.

Completion

▸
The output text generated by an LLM in response to a prompt.

The model "completes" the text sequence based on its training. Can be text, code, structured data, or function calls depending on the prompt and model capabilities.

Chain-of-Thought (CoT)

▸
Prompting technique that improves reasoning by asking the model to show its work step by step.

Adding "think step by step" or "show your reasoning" dramatically improves accuracy on math, logic, and complex tasks. Now built into many models natively.

Constitutional AI

▸
Anthropic's approach to AI alignment using a set of principles the model follows.

Instead of relying solely on RLHF, the model is trained to evaluate its own outputs against a "constitution" of principles. Results in more consistent and principled behavior.

Hallucination

▸
When AI generates false, fabricated, or unsupported information with apparent confidence.

A fundamental challenge with LLMs. Mitigated by grounding (RAG), temperature control, and explicit instructions to say "I don't know." Critical to detect in production.

Grounding

▸
Connecting an LLM to factual data sources to reduce hallucinations and improve accuracy.

RAG is the most common grounding technique. Other methods include search augmentation, database lookups, and fact-checking chains. Essential for enterprise applications.

Prompt Injection

▸
Attack where malicious input manipulates an LLM to ignore its instructions or reveal hidden prompts.

Can cause the model to bypass safety rules, leak system prompts, or perform unintended actions. Requires input validation, output filtering, and defense-in-depth strategies.

Jailbreak

▸
Technique to bypass an LLM's safety guardrails and elicit prohibited outputs.

Attackers craft prompts that trick the model into ignoring its training restrictions. Different from prompt injection (which targets the application layer). Continuously evolving cat-and-mouse game.

RAG & Data

RAG (Retrieval-Augmented Generation)

▸
Architecture that connects an LLM to external data sources to provide accurate, up-to-date answers.

Three steps: (1) convert documents to embeddings, (2) retrieve relevant chunks based on query, (3) feed chunks to LLM as context. The standard pattern for enterprise AI knowledge systems.

Embedding

▸
A numerical vector representation of text that captures its semantic meaning.

Similar texts produce similar vectors. Generated by specialized embedding models. Stored in vector databases for fast similarity search. Typical dimensions: 768 to 3072.

Vector Database

▸
Database optimized for storing and searching embedding vectors at scale.

Examples: Pinecone, Weaviate, Qdrant, pgvector. Uses approximate nearest neighbor (ANN) algorithms for fast similarity search. Core infrastructure for RAG systems.

Semantic Search

▸
Finding information based on meaning and intent rather than exact keyword matching.

Uses embeddings to compare query meaning against document meanings. Returns conceptually relevant results even when different words are used.

Chunking

▸
Breaking documents into smaller, manageable pieces for RAG processing.

Chunk size affects retrieval quality — too small loses context, too large dilutes relevance. Strategies include fixed-size, sentence-based, and semantic chunking. Typical size: 256–1024 tokens.

Context Window Management

▸
Strategies for fitting the most relevant information into an LLM's limited context.

Includes prompt compression, dynamic context selection, summarization of long documents, and priority-based inclusion. Critical when context exceeds window limits.

Retrieval

▸
The process of finding and fetching relevant documents or data chunks from a knowledge base.

Uses semantic search, keyword search, or hybrid approaches. Retrieval quality directly determines RAG output quality. Measured by precision and recall metrics.

Knowledge Base

▸
A structured collection of documents, data, and information that an AI system can search and reference.

Can include PDFs, wikis, databases, APIs, and other data sources. Requires regular updates and quality maintenance. The "brain" behind RAG systems.

Evaluation & Metrics

RAGAS (RAG Assessment Score)

▸
Framework for evaluating RAG pipeline quality using automated metrics.

Measures faithfulness, answer relevancy, context precision, and context recall. Provides quantitative scores without manual review. Industry standard for RAG evaluation.

Faithfulness

▸
RAGAS metric measuring whether the generated answer is supported by the retrieved context.

Score 0–1. High faithfulness means the answer doesn't contain claims unsupported by the provided documents. Detects hallucinations in RAG outputs.

Answer Relevancy

▸
RAGAS metric measuring how well the generated answer addresses the original question.

Score 0–1. Penalizes incomplete answers and off-topic content. A high score means the response directly answers what was asked.

Context Precision

▸
RAGAS metric measuring whether retrieved chunks are relevant to the question.

Score 0–1. High precision means the retrieval step found the right documents. Low precision indicates the search is returning irrelevant content.

Context Recall

▸
RAGAS metric measuring whether all necessary information was retrieved.

Score 0–1. High recall means no important information was missed. Low recall indicates gaps in the knowledge base or retrieval configuration.

Golden Dataset

▸
A curated set of question-answer pairs used as ground truth for evaluation.

Created by domain experts. Contains expected answers for known questions. Used to benchmark RAG pipeline changes and catch regressions. Typically 50–200 pairs.

Baseline

▸
The initial performance measurement against which improvements are compared.

Established before making changes. Includes metrics like accuracy, latency, and cost. Without a baseline, you can't prove whether changes actually improved the system.

Accuracy

▸
The percentage of correct or acceptable outputs produced by an AI system.

Measured against ground truth or human evaluation. Can be broken down by category, difficulty, or domain. The most intuitive but sometimes misleading metric.

Bias / Fairness

▸
Whether an AI system treats different groups or inputs equitably.

Bias can be in training data, model weights, or application logic. Testing includes checking outputs across demographics, languages, and edge cases. Regulatory requirement in many jurisdictions.

Quality Gate

▸
A checkpoint in the development pipeline where output must meet defined quality standards to proceed.

Includes automated metric thresholds (e.g., RAGAS faithfulness > 0.8) and manual review stages. Prevents low-quality models from reaching production.

Security & Red Teaming

Red Teaming

▸
Systematically testing AI systems by attempting to make them fail, produce harmful outputs, or behave unexpectedly.

Involves adversarial prompting, edge case testing, and social engineering attempts. Required before production deployment. Results feed into guardrail improvements.

Prompt Injection

▸
Attack where crafted input causes an LLM to deviate from its intended instructions.

Direct injection inserts malicious instructions in user input. Indirect injection hides instructions in external data the LLM processes. Defense requires input sanitization and output validation.

PII Detection (Personally Identifiable Information)

▸
Identifying and protecting personally identifiable information in AI inputs and outputs.

Includes names, emails, phone numbers, addresses, ID numbers. Automated detection uses regex patterns and NER models. Required for GDPR compliance and data protection.

Guardrails

▸
Safety mechanisms that constrain AI behavior within acceptable boundaries.

Input guardrails filter dangerous prompts. Output guardrails check responses for harmful content. Process guardrails limit agent actions. Can be rule-based or AI-powered.

Input Validation

▸
Checking and sanitizing user inputs before they reach the AI model.

Includes length limits, content filtering, format validation, and injection detection. First line of defense against adversarial attacks. Should never be the only defense.

Threat Modeling

▸
Systematic process of identifying potential security threats and vulnerabilities in an AI system.

Maps attack surfaces, identifies threat actors, evaluates risk levels, and prioritizes mitigations. Should cover data poisoning, model extraction, and prompt attacks.

Data Classification

▸
Categorizing data by sensitivity level to determine appropriate handling and protection.

Levels typically include Public, Internal, Confidential, and Restricted. Determines which data can be used for training, stored in vector databases, or processed by LLMs.

Adversarial Testing

▸
Deliberately crafting inputs designed to cause AI system failures or unexpected behavior.

Goes beyond normal testing to explore edge cases, boundary conditions, and attack scenarios. Includes automated fuzzing and manual creative attacks.

OWASP Top 10 for LLMs

▸
Industry-standard list of the most critical security risks for LLM applications.

Includes prompt injection, insecure output handling, training data poisoning, model denial of service, supply chain vulnerabilities, and more. Essential reference for AI security teams.

Governance & Compliance

EU AI Act

▸
European Union regulation classifying AI systems by risk level with corresponding compliance requirements.

Four tiers: Unacceptable (banned), High-risk (strict requirements), Limited risk (transparency), Minimal risk (no rules). Effective 2025–2026. First comprehensive AI law globally.

Risk Classification

▸
Categorizing AI systems by potential impact level to determine governance requirements.

Considers factors like autonomy, affected population, domain sensitivity, and reversibility of decisions. Higher risk = more documentation, testing, and human oversight required.

FRIA (Fundamental Rights Impact Assessment)

▸
Formal assessment of how an AI system may affect individuals' fundamental rights.

Required for high-risk AI under EU AI Act. Evaluates impact on privacy, non-discrimination, freedom of expression, and human dignity. Must be completed before deployment.

GDPR (General Data Protection Regulation)

▸
EU regulation governing the collection, processing, and storage of personal data.

Requires lawful basis for processing, data minimization, right to erasure, and breach notification. Directly impacts what data can be used in AI training and RAG systems.

Audit Logging

▸
Recording all AI system actions, decisions, and data access for compliance review.

Logs must be tamper-proof, timestamped, and retained per policy. Enables post-incident investigation, compliance audits, and performance monitoring.

Record-keeping

▸
Maintaining documented evidence of AI system development, testing, and deployment decisions.

EU AI Act requires technical documentation, risk assessments, test results, and change logs. Must be available for regulatory inspection.

Transparency

▸
Making AI system capabilities, limitations, and decision-making processes understandable to users and stakeholders.

Includes disclosing when AI is being used, explaining how decisions are made, and providing information about training data and known limitations.

Explainability

▸
The ability to describe how and why an AI system produced a specific output or decision.

Ranges from simple feature importance to detailed reasoning traces. Required for high-risk applications. Chain-of-thought prompting helps but isn't sufficient for full explainability.

Human-in-the-Loop (HITL)

▸
System design where humans review, approve, or override AI decisions at critical points.

Required for high-risk applications under EU AI Act. Defines escalation triggers, review workflows, and override authority. Balances automation efficiency with human judgment.

Escalation

▸
The process of routing an AI decision or situation to a human operator when predefined thresholds are exceeded.

Triggers include low confidence scores, sensitive topics, high-value decisions, or explicit user requests. Must have clear SLAs for response time.

KAF Framework

KAF (Kyndryl Agentic AI Framework)

▸
Kyndryl's enterprise platform for building, deploying, and governing AI agent systems.

Five-layer architecture covering data ingestion, agent building, AI core, policy enforcement, and human oversight. Designed for regulated industries requiring full auditability.

L1 — Data Ingestion

▸
KAF's first layer handling data collection, normalization, and preparation for AI processing.

Includes connectors to enterprise data sources, ETL pipelines, data quality checks, and format standardization. Feeds clean data into the embedding and retrieval layers.

L2 — Agent Builder

▸
KAF's second layer providing tools and interfaces for creating and configuring AI agents.

Includes prompt templates, tool registries, workflow designers, and testing sandboxes. Enables both developers and business users to build agents within governance guardrails.

L3 — AI Core

▸
KAF's third layer managing LLM interactions, orchestration, and core AI processing.

Handles model selection, prompt routing, response generation, and multi-agent coordination. Abstracts away provider-specific APIs behind a unified interface.

L4 — Policy-as-Code

▸
KAF's fourth layer enforcing governance rules programmatically rather than through manual processes.

Policies are written as executable code that automatically checks compliance. Includes data classification rules, output filters, access controls, and audit requirements.

L5 — Human-in-the-Loop

▸
KAF's fifth layer managing human oversight, review workflows, and escalation processes.

Defines when human approval is needed, routes decisions to appropriate reviewers, and tracks approval history. Configurable per use case and risk level.

Agent Registry

▸
A centralized catalog of all AI agents in an organization, their capabilities, status, and ownership.

Enables discovery, reuse, and governance of agents. Tracks versions, performance metrics, and compliance status. Prevents agent sprawl and duplication.

Connectors

▸
Pre-built integrations that link KAF to external enterprise systems and data sources.

Examples include ServiceNow, SAP, Salesforce, databases, and cloud storage. Standardize data access patterns and enforce security policies at the integration layer.

Architecture & DevOps

CI/CD (Continuous Integration / Continuous Deployment)

▸
Automated pipeline that builds, tests, and deploys code changes continuously.

CI merges and tests code frequently. CD automatically deploys approved changes to production. Essential for AI systems where model updates and code changes happen frequently.

Docker

▸
Platform for packaging applications and their dependencies into portable containers.

Ensures consistent behavior across development, testing, and production environments. Standard for deploying AI services. Images can include model weights, dependencies, and application code.

Microservices

▸
Architecture pattern where an application is built as a collection of small, independent services.

Each service handles one function (e.g., embedding, retrieval, generation). Enables independent scaling, deployment, and technology choices. Communicates via APIs.

Health Check

▸
Automated endpoint that reports whether a service is running correctly and ready to handle requests.

Returns status of dependencies (database, model, external APIs). Used by load balancers and orchestrators to route traffic and trigger alerts.

Blue-Green Deployment

▸
Deployment strategy using two identical production environments to enable zero-downtime releases.

"Blue" runs current version, "Green" runs new version. Traffic switches instantly after validation. Enables immediate rollback by switching back. Common for AI model updates.

Rollback

▸
Reverting a deployment to a previous known-good version when issues are detected.

Must be fast (seconds, not minutes) and automated. Requires keeping previous versions available. Critical for AI systems where model updates can introduce subtle regressions.

Staging

▸
A pre-production environment that mirrors production for final testing before release.

Should use production-like data volumes, configurations, and infrastructure. AI systems should test with representative queries and edge cases in staging.

Load Balancing

▸
Distributing incoming requests across multiple service instances to optimize performance and reliability.

Prevents any single instance from becoming overwhelmed. Strategies include round-robin, least-connections, and weighted distribution. Essential for AI inference endpoints.

Latency

▸
The time delay between sending a request and receiving a response.

For AI systems, includes network time, queue time, and inference time. Measured in milliseconds. User-facing applications typically target under 2 seconds. Streaming helps perceived latency.

Throughput

▸
The number of requests a system can process per unit of time.

Measured in requests per second (RPS) or tokens per second. Affected by model size, hardware, and batching. Key metric for capacity planning and cost estimation.

Uptime / SLA (Service Level Agreement)

▸
The percentage of time a service is operational, often guaranteed by a Service Level Agreement.

99.9% uptime allows ~8.7 hours downtime per year. 99.99% allows ~52 minutes. SLAs define penalties for failing to meet targets. AI systems must account for model provider outages.

Incident Response

Capability Incident

▸
An incident where an AI agent fails to perform its intended function correctly.

Examples: model returns wrong answers, tool calls fail, latency exceeds thresholds. Root cause is typically model degradation, data quality issues, or infrastructure failures.

Coordination Incident

▸
An incident where multiple agents or systems fail to work together properly.

Examples: handoff failures, deadlocks, conflicting actions, data inconsistency between agents. Often caused by schema changes, timing issues, or missing error handling.

Drift Incident

▸
An incident where AI system behavior gradually degrades over time without an obvious trigger.

Caused by changing data distributions, evolving user patterns, or upstream model updates. Detected through continuous monitoring and metric tracking. Requires retraining or recalibration.

Root Cause Analysis (RCA)

▸
Systematic investigation to identify the fundamental reason an incident occurred.

Uses techniques like "5 Whys," timeline reconstruction, and fault tree analysis. Distinguishes symptoms from causes. Must identify both technical and process failures.

Post-Incident Review (PIR)

▸
A structured meeting held after an incident to analyze what happened and prevent recurrence.

Blameless format focusing on system and process improvements. Documents timeline, impact, root cause, and action items. Also called "retrospective" or "post-mortem."

Blast Radius

▸
The scope of impact when a failure or incident occurs in a system.

Smaller blast radius = fewer users/systems affected. Design principles include isolation, circuit breakers, and feature flags to limit blast radius. Key factor in risk assessment.

Containment

▸
Immediate actions taken to stop an incident from spreading or causing further damage.

Examples: disabling a malfunctioning agent, blocking traffic to an affected endpoint, reverting to a backup model. Speed is critical — contain first, investigate later.

Hotfix

▸
An urgent code change deployed directly to production to resolve a critical issue.

Bypasses normal release cycles but should still go through minimal testing. Must be backported to the main development branch. Document the emergency process used.

Runbook

▸
A documented set of step-by-step procedures for handling specific operational scenarios.

Covers routine operations (deployments, scaling) and incident response. Should be tested regularly and updated after incidents. Reduces dependency on individual expertise.

Playbook

▸
A strategic guide describing how to respond to a specific type of incident or scenario.

Higher-level than a runbook — a playbook defines roles, communication plans, decision criteria, and escalation paths. Runbooks are the step-by-step procedures within a playbook. Organizations maintain playbooks for common incident types (model drift, data breach, outage).

Anti-pattern

▸
A common but counterproductive approach that initially seems helpful but causes problems.

AI anti-patterns include: deploying without evaluation, skipping human review for high-risk outputs, using production data in development without masking, and building monolithic agent systems.

Business & Project Management

Discovery

▸
The initial project phase focused on understanding requirements, constraints, and feasibility before building.

Includes stakeholder interviews, data assessment, technical evaluation, and risk analysis. Outputs a recommendation on whether and how to proceed. Typically 2–4 weeks.

Go/No-Go Gate

▸
A formal decision point where stakeholders decide whether a project should proceed to the next phase.

Based on defined criteria including technical feasibility, business case, risk assessment, and resource availability. Documented with rationale. Prevents wasted investment.

Business Case / ROI (Return on Investment)

▸
A financial justification showing the expected return on investment for an AI initiative.

Compares costs (development, infrastructure, operations) against benefits (time saved, errors reduced, revenue increased). Must include both quantitative metrics and qualitative value.

Feasibility

▸
Assessment of whether a proposed AI solution can realistically be built and deployed with available resources.

Covers technical feasibility (can it be done?), operational feasibility (can it be maintained?), and economic feasibility (is it worth doing?). Part of the Discovery phase.

Stakeholder

▸
Any person or group with an interest in or influence over an AI project's outcome.

Includes sponsors, end users, IT operations, legal/compliance, data owners, and affected communities. Stakeholder management is critical for AI adoption and change management.

Scope

▸
The defined boundaries of what an AI project will and will not deliver.

Clearly documented in project charter. Includes features, data sources, user groups, and performance targets. Scope creep is a major risk in AI projects due to "it could also do..." thinking.

Sprint Retrospective

▸
A team meeting at the end of a sprint to reflect on what worked, what didn't, and what to improve.

Part of Agile methodology. Focuses on process improvement, not blame. Action items should be specific and tracked. Essential for continuous improvement in AI development.

Managed Services

▸
Outsourcing the operation and maintenance of IT systems or AI solutions to a service provider.

Provider handles monitoring, updates, scaling, and incident response per agreed SLAs. Kyndryl's core business model. Enables clients to focus on business value rather than operations.

AI Innovation Lab Method

▸
Kyndryl's structured approach for rapidly prototyping and validating AI use cases with clients.

Combines Design Thinking with technical feasibility assessment. Typically 3–5 day intensive workshops producing a validated prototype and business case. Bridges the gap between idea and implementation.
